# 厨房实时任务调度系统MVP文档

## 文档信息
- **项目名称**: 厨房实时任务调度系统
- **文档版本**: MVP v1.0
- **创建时间**: 2026年
- **目标**: 实现厨房数字化管理的最小可行产品

---

## 1. MVP愿景与目标

### 1.1 项目愿景
打造新一代厨房实时任务调度系统，实现从传统纸质订单到数字化厨房管理的无缝转型，提升餐厅运营效率和服务质量。

### 1.2 核心目标
- ✅ **效率提升**: 订单处理速度提升60%以上
- ✅ **错误降低**: 减少人为失误率至1%以下  
- ✅ **实时同步**: 厨房各岗位信息实时共享
- ✅ **基础数据**: 提供简单的经营数据统计

## 2. MVP功能范围

### 2.1 核心必做功能 ✅

## 3. 技术架构MVP版

### 3.1 简化技术栈
```
前端: Vue3 + 基础响应式设计 + ArcoDesign(不强制使用)或 Tailwind CSS
后端: NestJS + RESTful API  
数据库: PostgreSQL (核心表结构)
实时通信: WebSocket基础推送
部署: Docker容器化
```

### 3.2 MVP数据库最小表结构

#### users（用户表）

```sql
id               -- 主键
name			-- 名称
tel			    -- 手机号
avatar			-- 头像
station          -- 工位
password		-- 登陆密码
```

#### orders (订单表)
```sql
id              -- 主键
hall_number     -- 台号（默认为[未指定]；如为打包，则写为[打包+用餐时间]）
people_count    -- 人数  
table_count     -- 桌数 
status          -- 状态(created → started → serving → urged → done → cancelled)
created_at      -- 创建时间
meal_time       -- 用餐时间（年月日时分，时分不在录入时填入，如为打包则自动解析台号并填入）⭐ 建议使用TIMESTAMP类型
meal_type       -- 用餐类型（午/晚）
start_time      -- 起菜时间（若为打包菜，start_time = meal_time）
remark          -- 备注
```

#### order_items (订单菜品表)
```sql
id              -- 主键
order_id        -- 订单ID
dish_id         -- 菜品ID
quantity        -- 单桌数量（可有一位小数）
weight          -- 重量（如5两，1斤）
status          -- 状态(pending → prep → ready → served → cancelled)
priority        -- 优先级
remark          -- 备注
served_at       -- 上菜时间
created_at      -- 创建时间
```

#### dishes (菜品表)
```sql
id              -- 主键
name            -- 菜品名称
station_id      -- 工位ID
category_id     -- 分类ID
shortcut_code   -- 模糊搜索代号
countable       -- 是否计数，比如托炉饼一份需按一桌的人数计数（如是，则数量=人数/桌数）
recipe_id       -- 对应菜谱
need_prep       -- 是否需要预处理，如否则跳过status.prep(默认为false)
is_active       -- 启用或禁用菜品
```

#### dish_categories (菜品分类表)
```sql
id              -- 主键
name            -- 类名
```

#### stations (工位表)
```sql
id              -- 主键
name            -- 工位名称
```

####  **recipes（菜谱）**

```sql
id				-- 主键
name			-- 菜品名称
description		 -- 描述文字
ingredient_list  -- 用料
images			-- 配图
steps			-- 做法
tags			-- 标签
```

## 4.出餐逻辑

在总览视图中，同名同状态同优先级菜品数量 = order_items.quantity × orders.table_count + 其它订单的order_items.quantity × orders.table_count + ...

若order_items.dish.countable = true, 需要在details中显示（如托炉饼共30个，其中有12个2份，6个1份，需用两行文字全部显示）

订单未起菜时，该订单的所有菜品的优先级为0。

订单未开始时，status = 'created'；订单进入用餐时间后，status = 'started'；

订单已起菜时，status = 'serving'；订单被暂停时，status = 'started'；

若某订单被催菜，该订单的status = 'urged'，然后在order_items的details ui中增加“[台号]催菜”；当status = 'urged'并在上了一道菜后，该订单的status = 'serving'，删除催菜备注。

红色卡片：优先出，优先级3；

黄色卡片：等一下，优先级2；

绿色卡片：不急，优先级1；

未起的菜品为灰色卡片，优先级0；

已出为-1，白色；

上菜顺序：started → 凉菜 → serving → 前菜 → 中菜、点心、蒸菜 → 后菜 → 尾菜  → done；

起菜后默认优先级：前菜，3级，中菜，2级；后菜1级，尾菜1级；该订单前面的整个分类中的菜上了之后，该订单后面的整个分类中的菜自动+1；

按照此订单的上一道菜间隔时间，超过[该订单菜品数量×150×餐厅当前时段订单总数×0.5]秒（可在设置中调校）自动调整优先级

起菜后后来加菜的菜品优先级为3级；

## 5. 用户界面设计

 对原型图的补充说明，需按照原型图来设计；使用MVP工具jishi-design来参考原型设计稿；

### 整体布局

**header**

左侧：员工头像（点击弹出侧边栏）+[打荷/切配/冷菜/蒸菜/点心] + 用户名

右侧：日期选择，午/晚餐切换

再加一排按钮：起菜、催菜、加菜、暂停、退菜、录入订单

点击起菜/催菜/暂停后，弹出一个表单容器（背景模糊），选择对应的台号（可多选），执行起菜/催菜/暂停操作，确认或取消；弹窗中也要有状态检查。

**body**

左侧菜单组件：总览置顶+订单选择区选项卡，使用台号区分订单；按上一道菜间隔时间排序

催菜的台号显示为红字加粗，在催的台号上了一道菜后取消催菜状态；若为未起菜或暂缓状态，台号文字颜色为灰色，可选中

底部Nav：烹调/库存/待办/历史
当前处于烹调板块，主要开发烹调板块

### **订单录入页面**

点击录入订单后，切换view；需要收集人数、桌数、台号、用餐时间（年月日）、用餐类型（午/晚）、菜品表；

点击菜品按钮选择菜品，选中后再次点击弹出一个表单容器（背景模糊），增减份量（使用+/-按钮，小于0即删除并关闭弹窗）、添加重量和备注信息；菜品顺序为上菜顺序；

该订单录入完成后，已选的菜品按钮保持已选状态（备注不保留），用于下一张订单的录入（若有选中，[取消]按钮变成[清空已选]按钮，点击后清空选中菜品并变回[取消]按钮）

若点击新增按钮，弹出一个表单容器（背景模糊），输入菜品名称（必填）、份量（选填）、重量（选填）、备注（选填）、是否按人头计数（必填），然后在菜品库中新增一个仅有菜品名的菜品；

点击加菜/退菜后，弹出一个容器，选择对应的台号，（复用订单录入弹窗中的菜品选择部分，其中菜品顺序按对应订单中已定下的菜品排序，已选的菜品按钮保持已选状态，排除已上菜品）新增或选择或取消选择菜品，然后确认或取消；

### 单个订单视图

该页面展示指定订单的所有信息。

首先是订单信息表格：人数，桌数，厅号，下单人，电话，负责人，电话，起菜时间（显示到时分；若为打包菜，显示文字改为[取餐时间]），上一道菜间隔时间，备注

其中下单人，电话，负责人，电话暂不显示

然后是[已出]菜品表格

再然后是[待上]菜品表格，按出餐顺序排序；

如有被取消的菜品，则显示[取消]表格；

最后是三个按钮：[添加备注]、[编辑订单信息]、[取消订单（二次确认）]

### 总览视图

该页面总览所有订单的菜品，自动合并同名同状态同优先级的菜品。

所有卡片文字均垂直居中；

菜品使用20px双列卡片展示（标注文字也是20px），第一行显示菜品名称（超出则省略中间的字，需展示最后一个字）和数量（如椒盐基围虾×2）；

已出菜品使用18px瀑布流卡片展示，优先级为-1，5秒内未操作自动折叠该视图；

未起菜（优先级为0）则在未起中显示。

待上菜品的其余行显示对菜品的标注，如12个·2份，微辣·1份；若有数量或重量，需显示所有的量；若有备注，则只需显示对应的备注；

举例说明，托炉饼共30个，其中有12个2份，6个1份，需用两行文字全部显示；水芹×2份，其中一份需清淡，则仅在第二行显示清淡·1份；

若同时remark中有分盘要求（如12 12 14），则显示为12个·2份，14个·1份，不使用数量显示

若status为待切配(pending)或待处理(prep)，对应菜品卡片加6px蓝色描边，同时居上覆盖提示待切配或待处理字样；

点击卡片状态流转，完成出菜或完成切配与处理；

切配完成后如需处理则自动在待办中新建对应事项；

长按某个菜品卡片（调整其出餐逻辑），弹出一个容器（背景模糊），可以增减其数量（减去的菜品优先级-1，但不得小于1；若为-1，则改为3级），或调整全部优先级（加急3级/放缓2级/再缓1级/出餐-1级）；在弹窗中添加一个查看菜谱按钮（先留空），若提示“未切配/未处理”则跳转到菜谱中的切配或打荷部分
