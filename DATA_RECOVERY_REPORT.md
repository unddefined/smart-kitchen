# 🚨 生产环境数据丢失事故报告与恢复记录

## 事故概述
- **发生时间**: 2026年2月28日
- **影响范围**: 生产环境数据库菜品数据从50+条减少到17条
- **根本原因**: GitHub Actions部署时执行了包含危险清空操作的seed脚本

## 问题诊断过程

### 1. 初始发现
通过SSH检查生产环境数据库状态：
```bash
ssh -i "C:/Users/66948/.ssh/PWA应用密钥.pem" root@8.145.34.30 "cd /root/smart-kitchen && docker compose exec postgres psql -U smart_kitchen -d smart_kitchen_prod -c 'SELECT COUNT(*) as dish_count FROM dishes;'"
```
结果显示只有17个菜品，远低于预期的50+个。

### 2. 根本原因分析
通过检查Git提交历史发现：
- 最近一次提交 `8561e0c` 修改了 `backend/prisma/seed.ts` 文件
- 该文件包含了危险的清空操作：
```typescript
// 清空现有数据
await prisma.orderItem.deleteMany();
await prisma.order.deleteMany();
await prisma.dish.deleteMany();
await prisma.recipe.deleteMany();
await prisma.dishCategory.deleteMany();
await prisma.station.deleteMany();
```

### 3. 数据源确认
根据项目中的 `菜品库.md` 文件确认了完整的菜品清单，共包含48个菜品。

## 恢复过程

### 1. 紧急响应
- 立即暂停所有自动化部署流程
- 创建多个恢复脚本方案

### 2. 数据恢复执行

## 预防措施

### 1. 代码层面改进
**已实施的修复**：
- 修改 `seed.ts` 文件，移除所有清空数据的操作
- 添加安全检查机制，避免覆盖现有生产数据
- 实现增量式数据创建逻辑
- 增加详细的执行日志

### 2. 部署流程优化
**建议实施的措施**：
```yaml
# 在GitHub Actions中添加数据保护检查
- name: Check production data safety
  run: |
    # 检查是否有危险的清空操作
    if grep -r "deleteMany()" ./backend/prisma/seed.ts; then
      echo "⚠️  Danger: seed script contains delete operations!"
      exit 1
    fi
    
    # 检查是否连接到生产环境
    if [ "$NODE_ENV" = "production" ]; then
      echo "🚫 Production environment detected - skipping seed execution"
      exit 0
    fi
```

### 3. 监控和告警
**建立以下监控机制**：
- 数据库记录数异常波动监控
- 关键表数据量变化告警
- 自动化数据完整性检查
- 定期数据备份验证

### 4. 操作规范
**制定生产环境操作准则**：
1. **禁止在生产环境执行seed脚本**
2. 数据变更必须通过专门的迁移脚本
3. 所有数据库操作前必须创建快照
4. 建立变更审批流程
5. 定期进行灾难恢复演练

## 经验教训

### 技术层面
1. **Seed脚本不应包含破坏性操作**：开发环境和生产环境应该有不同的数据初始化策略
2. **需要环境隔离机制**：通过环境变量控制脚本行为
3. **建立数据保护机制**：关键业务数据应该有只读保护

### 流程层面
1. **代码审查的重要性**：涉及数据操作的变更需要特别严格的审查
2. **自动化测试的必要性**：应该有测试验证数据操作的安全性
3. **文档和培训**：团队成员需要清楚了解生产环境操作规范

### 监控层面
1. **实时监控的重要性**：及时发现异常变化
2. **告警机制**：关键指标异常时能够及时通知
3. **审计日志**：记录所有重要的数据变更操作
